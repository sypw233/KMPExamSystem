# KMPExamSystem 未完成功能详细开发计划表

本计划表针对 KMPExamSystem 当前遗留的关键功能（UI 层约 39% 缺口）进行了深度拆解，按照重要性与业务约束关系，划分为四个阶段推进。

---

## 阶段一：核心业务闭环（高优先级 / 阻断性功能）
> 目标：打通教师「组卷」→学生「查看解析」→教师「评分」的完整业务链路。

### 1。考试选题组卷 UI（⭐ 极高优先级）
*   **功能描述**：教师在新建考试后，需要向考试中添加具体题目。没有该功能，学生面对的是空泛考试，流程无法跑通。
*   **前置条件**：考试信息已成功建立（处于「草稿」状态）。
*   **涉及 API**：
    *   获取候选题目列表：通过现有的 `QuestionApi.getQuestions(courseId)`。
    *   向考试添加/移除题目接口（*说明：现有后端代码中若是考试和题目属于多对多关系，可能需要 `ExamApi.addQuestionToExam(examId, questionId)` 等相关 API*）。
*   **页面设计规划**：
    *   **入口**：在 `TeacherExamManageScreen` 的草稿考试卡片上，新增「组卷」按钮。
    *   **UI 组件**：设计全屏弹窗或子页面 `ExamComposeScreen`。左侧（大屏平台）/ 上部（手机端）显示待选题目（支持类型过滤和搜索），右侧/下部显示已选题目列表及总分统计。
    *   **交互**：点击题目左侧 `Checkbox` 勾选，自动统计目前总分，到达考试设定总分时给予提示。

### 2。主观题评分界面与 AI 批卷（⭐ 极高优先级）
*   **功能描述**：现阶段如果存在简答题，目前学生端提交后无法获得分数。教师需要一个页面来阅读学生的回答并打分。
*   **涉及 API**：
    *   获取考试所有提交记录：`SubmissionApi.getExamSubmissions(examId)`
    *   获取单次提交内解答情况：`SubmissionApi.getSubmissionDetail(submissionId)`
    *   执行评分并保存：`SubmissionApi.gradeSubmission(submissionId, grades)`
    *   请求 AI 评分：`AiGradingApi.aiGrade(request)` 给出分数建议与点评。
*   **页面设计规划**：
    *   **阅卷列表页**：展示参加该考试的所有学生及其提交状态、客观题得分。
    *   **详情批阅页**：
        1. 屏幕上半部分显示核心题目要求和标准答案。
        2. 下半部分显示学生回答。
        3. 底栏/侧边栏提供分数滑块/轮盘或文本框录入。
        4. 提供**「AI 阅卷建议」**悬浮按钮，请求 `AiGradingApi` 填充分数及建议评语，再由教师点击确认提交。

### 3。学生成绩详情解析页（⭐ 极高优先级）
*   **功能描述**：学生完成考试或成绩发布后，需点进成绩历史查看错题和正确解析。
*   **涉及 API**：`SubmissionApi.getSubmissionDetail`（现有返回结构若含正确答案即可直接适配，无的话需要扩展响应 DTO）。
*   **页面设计规划**：
    *   **入口**：`GradeHistoryScreen` 中每条考试记录新增点击事件，导航进入 `GradeDetailScreen`。
    *   **UI**：极简复用考场页 (`ExamTakingScreen`) 组件。差异在于：
        *   屏蔽交互操作（不可切题）。
        *   绿色高亮正确选项，标红选错选项。
        *   主观题展示教师批语/AI 评分。

## 阶段二：数据展示与系统管理完善（中高优先级）
> 目标：将底层做好的大量 API（题库、统计等）对接到呈现层。

### 1。题库管理界面（Question Bank UI）
*   **功能描述**：将分散的题目按知识体系收集为可重用的「题库」（Repository API 已完整，完全缺少 UI）。
*   **涉及 API**：`QuestionBankApi` 全套接口（增删查改题库，题库加题删题，导出题库）。
*   **页面设计规划**：
    *   在教师左侧导航栏加一个「全部题库」入口（或者在题目管理的 Tab 栏）。
    *   列表以卡片排列，显示题库覆盖科目及含题量。
    *   点击题库名进入题库详情（含题列表）。

### 2。系统与统计仪表盘（Admin Dashboard & Visuals）
*   **功能描述**：目前管理员与普通学生看着极为相似的欢迎页。管理员需要数据大屏支持。
*   **涉及 API**：`StatisticsApi` 全套接口（`getSystemOverview`, `getCourseStatistics` 等）。
*   **页面设计规划**：
    *   引入 Compose 适配的轻量图表库（如 `YCharts`、`Compose-Charts`）。
    *   管理员的 `DashboardScreen` 完全替代为基于格栅的统计卡片页面。包含系统资源情况（Pie chart）、近日活跃用户数（Line chart）、通过率漏斗等。

### 3。AI 配置与系统管理设置
*   **功能描述**：管理员调校 AI 大语言模型接口凭证和 Prompt 配置等。
*   **涉及 API**：`AiGradingApi.getAiConfigs`, `AiGradingApi.updateAiConfig`。
*   **页面设计规划**：
    *   新建 `SystemSettingsScreen`（管理员可见）。
    *   支持切换当前使用的大模型平台架构，密文方式录入 API_KEY、调整 Temperature 及预设 Prompt 模版。

## 阶段三：个人事务体验强化（中优先级）
> 目标：补齐普通单薄的 C 端用户（学生）使用体验的缺失。

### 1。个人资料修改与密码管理
*   **涉及模块**：
    *   `ProfileScreen` 扩建表单。
    *   调用 `UserManageApi` 或专属 User Auth 更新账户（需确认后端是否有向学生开放的 `/api/users/me` 更新接口，否则需通过 Admin 接口赋权或新开端点）。
*   **实现路径**：
    *   提供包含原有信息占位文本对的编辑模式。
    *   独立的重置密码（输入旧密码、新密码、二次确认）弹窗。

### 2。选课反向解绑（退选）与前台考试分发
*   **功能描述**：学生退选课程；点击课程详情页能看到它绑带的具体考试。
*   **涉及 API**：若 `CourseApi.removeStudentFromCourse` 支持学生身份自我调用即可直接使用。
*   **实现路径**：在在已选课程右侧添加「退出课程」红色危险按钮，双重确认提示。

### 3。监考事件的端侧检测上报
*   **功能描述**：监测学生切屏等作弊可能迹象并上报给监考方系统。
*   **涉及 API**：`SubmissionApi.recordProctoringEvent`。
*   **页面设计规划**：
    *   挂载全局生命周期或窗口焦点观测器 (`WindowFocusObserver`) 给 `ExamTakingScreen`。
    *   失去焦点超过 N 毫秒，则记录事件（失焦毫秒数及次数），重登进入则上报接口。
    *   达到作弊阈值次，向 UI 发出强制交卷警告并断开连接。

## 阶段四：导入导出等基础设施（相对低优先级）
> 目标：完成外部与内部生态数据交融。

### 1。导出与导入组件化
*   **子任务 1 成绩导出**：教师可对批改完的指定考试记录导出成绩报表。
*   **子任务 2 题库与文件的批量导入出**：支持从 CSV / 模板 Excel 导入系统，或者反之导出归档。
*   **技术挑战**：
    *   涉及在 KMP 开发模式下的多平台文件读写方案。
    *   Android 上需要调用 SAF 分享及存储；Desktop 平台需要 `JFileChooser` / 第三方窗口来打开原生系统对话框，以选取或保存文件。
    *   需要对接 `FileApi`。

### 2。通知系统的全系统广播能力
*   **功能描述**：管理员可以发布通知事件，推送给全量用户池。
*   **实现路径**：
    *   在管理员配置增加「发布新通告」按钮，使用富文本形式写入。
    *   后端记录一条群体 broadcast 发送池，当活跃端请求拉取最新通知时分发给他们，或结合 websocket 提供即时弹出气泡体验（如果有后端支持的话）。
